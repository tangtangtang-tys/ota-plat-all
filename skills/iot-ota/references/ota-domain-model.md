# OTA 领域模型

处理 OTA 升级管理系统相关任务时，优先读取本文件。除非用户提供了更新需求，否则把这里的规则作为当前项目的领域基线。

如果任务涉及 OTA2.0 的中心-边缘架构、执行链路、队列、Redis、区域聚合、taskId 隔离、候选认领等技术规则，还需要继续读取 [ota2-architecture.md](ota2-architecture.md)。

## 核心概念

| 名词 | 说明 |
| --- | --- |
| OTA 升级 | 对已部署物联网设备进行远程固件升级。 |
| 升级任务 | 一次针对特定设备集合的固件升级操作，包含目标版本、升级策略、执行区域、任务周期等配置。 |
| 整包升级 | 设备下载完整目标固件包。兼容性好，适合多数常规升级和跨版本升级场景。 |
| 差分升级 | 设备仅下载当前版本与目标版本之间的差异包。包体积小，但要求设备当前版本与差分包基线版本严格匹配。 |
| 升级策略 | 确定目标设备范围的方式，包括指定版本号升级、文件导入升级、手动导入升级。 |
| 任务周期 | 升级任务计划执行的开始和结束时间。开始时间必须晚于当前时间，周期不能超过 7 天。 |
| 产线负责人审批 | 指定版本号升级和文件导入升级需要审批；手动导入升级无需审批，发布后直接进入待执行或分发流程。 |
| 活跃设备扫描 | OTA2.0 中指定版本号升级的执行方式，边缘按机型扫描 Redis 活跃设备，并在执行时评估当前版本和策略。 |
| 候选认领 | OTA2.0 中导入设备创建时无法识别归属 regionId 时的处理方式。设备上线后，由真实所在边缘节点向中心认领。 |
| 区域聚合 | 中心侧按 regionId 保存的任务统计结果。设备执行明细保存在边缘。 |

## 硬性规则

- 差分升级目前仅 4G 模组产线支持。
- 差分升级要求目标版本和基线版本都具备可用于差分的文件。
- 差分包必须已经生成，且状态为生成成功。
- 差分升级只能覆盖当前版本与差分包基线版本严格匹配的设备。
- 当产线支持、基线版本或差分包可用性不确定时，优先建议整包升级。
- 任务开始时间必须晚于当前时间。
- 任务结束时间必须晚于开始时间。
- 任务周期不能超过 7 天。
- 指定版本号升级和文件导入升级需要审批后执行。
- 手动导入升级无需审批。
- 审批驳回后的任务不可编辑、不可重新提交；需要重新创建任务。
- 结束或中止任务不可恢复。
- 早期操作指南规则：结束任务后，未开始升级的设备不再下发，已开始升级的设备继续完成并上报成功或失败。
- OTA2.0 技术方案规则：中止任务会把待升级和升级中设备置为失败，并清理候选、运行态和执行去重。
- 因为新旧文档对“结束/中止”语义不同，写需求、手册、测试或评审时必须显式确认当前采用哪一种产品口径。
- 已完成、已结束、已驳回通常为终态，不可再流转。
- OTA2.0 中心不保存设备执行明细，只保存目标清单和区域聚合；设备执行明细保存在边缘。
- OTA2.0 的评估、策略、统计和数量限制按 taskId 隔离；同一设备仍然互斥，不能同时执行多个 OTA 任务。
- OTA2.0 文件导入和手动导入不应要求用户选择集群或节点，由后台识别设备 regionId。
- OTA2.0 无法识别归属的导入设备作为候选记录，设备上线后由真实边缘节点认领。

## 升级策略

| 策略 | 设备选择方式 | 典型场景 | 是否审批 |
| --- | --- | --- | --- |
| 指定版本号升级 | 按设备当前固件版本筛选。支持全部升级、指定版本、排除版本。 | 大规模版本统一、按版本分批升级。 | 需要 |
| 文件导入升级 | 上传 CSV 或 Excel 设备标识清单。 | 外部系统导出清单、运营活动批次、指定客户设备。 | 需要 |
| 手动导入升级 | 页面逐行输入或粘贴设备标识。 | 小批量测试、定向修复、精确设备升级。 | 不需要 |

OTA2.0 补充规则：

- 指定版本号升级本质上是活跃设备扫描，由用户选择当前大区下的 regionId 范围或整个大区。
- 文件导入和手动导入由中心调用日志平台补充机型/版本，并调用设备归属接口补充 regionId。
- 已知归属但属于其他大区的设备，不进入当前中心任务。
- 未知归属的导入设备会作为候选记录广播到当前大区节点，设备上线后再认领。
- 文件/手动任务如果没有当前中心可下发的目标设备，发布失败。

## 版本过滤规则

| 规则 | 行为 |
| --- | --- |
| 全部升级 | 纳入所有低于目标版本的设备，并自动排除已是目标版本的设备。差分升级时，仅纳入存在匹配差分基线的版本。 |
| 指定版本 | 仅纳入手动勾选的源版本设备。 |
| 排除版本 | 排除手动勾选的版本，其余符合条件的版本纳入升级范围。 |

## 校验规则

任务配置需要校验：

- 任务名称已填写；如系统要求唯一，则不能重复。
- 已选择升级策略。
- 已选择升级包类型。
- 目标固件版本存在，并且符合整包或差分包的可选范围。
- 已选择任务区域或 regionId 范围；文件/手动导入在 OTA2.0 中由后台识别归属。
- 任务周期合法。
- 源版本号和目标版本号不能相同。
- 至少存在一个有效目标设备。

导入或手动输入设备需要校验：

- 设备标识长度为 1 到 64 个字符。
- 设备标识只允许字母、数字、中划线、下划线。
- 设备在平台中存在。
- 设备属于当前任务可操作的大区范围。
- 重复设备自动去重，并提示去重数量。
- 设备当前版本不是目标版本。
- 设备机型或硬件与目标固件兼容。
- 差分升级时，设备当前版本存在匹配的差分包基线。

## 任务状态流转

除非用户提供新版规则，否则使用下面的基础状态模型：

```text
指定版本号/文件导入任务发布 -> 待审批 -> 审批通过 -> 待执行 -> 执行中 -> 已完成
                                  |                              |
                               已驳回                         已结束/中止

手动导入任务发布 -------------------------------> 待执行 -> 执行中 -> 已完成
                                                        |         |
                                                   已结束/中止  已结束/中止
```

状态说明：

- 待审批：任务已提交审批，任务配置通常锁定不可编辑。
- 已驳回：审批人拒绝任务。终态。
- 待执行：任务已通过审批或无需审批，等待开始时间或等待边缘执行。
- 执行中：任务已经开始下发或评估设备。
- 已完成：所有目标设备返回终态，或任务周期结束后剩余设备按规则处理。
- 已结束/中止：操作人员主动结束或中止任务。终态，且不可恢复。
- 已失效：如果产品版本使用该状态，表示任务审批或执行窗口已经失效。终态。

## 监控语义

- 执行中任务应自动刷新，例如每 15 秒刷新一次。
- 非执行中任务可不自动刷新，支持手动刷新即可。
- 基础进度可按 `(升级成功数 + 升级失败数) / 设备总数 * 100%` 计算。
- 设备状态至少包括：待升级、升级中、升级成功、升级失败、已终止或失败/中止。
- 失败处理需要保留失败原因，并支持筛选失败设备，用于排查和重新创建任务。
- OTA2.0 中心只保存区域聚合统计；页面查看设备明细时，由中心按 regionId 转发到边缘查询。
- 边缘明细查询失败不等于设备升级失败，应保留上次统计并提示查询异常。

## 专家评审重点

作为物联网 OTA 产品专家，需要始终检查：

- 设备范围是否准确：产线、大区、regionId、源版本、目标版本、导入清单。
- 升级包是否兼容：整包/差分、差分基线、硬件兼容性。
- 风控是否充分：审批、预览、灰度、批量上限、手动结束/中止、失败重试。
- 状态是否一致：任务状态、终态、设备状态、允许操作。
- 审计是否完整：操作日志、审批日志、任务配置快照、失败原因。
- 用户体验是否清晰：校验提示、发布前预览、不可逆操作确认。
- OTA2.0 架构是否匹配：中心-边缘数据边界、区域明细查询路径、taskId 隔离、设备互斥、候选认领、数量上限口径。
- 语义是否一致：“结束任务”采用旧操作指南口径，还是采用 OTA2.0 中止口径。
