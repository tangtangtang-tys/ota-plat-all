# OTA2.0 架构参考

当任务涉及 OTA2.0 架构、技术可行性、技术评审、工作流程评审、测试设计，或产品规则受到中心-边缘执行链路影响时，读取本文件。

## 架构总览

OTA2.0 采用中心-边缘架构：

| 角色 | 职责 |
| --- | --- |
| 前端控制台 | 创建、发布、中止任务；查看目标区域概览和设备明细。 |
| 大区中心 API | 提供前端接口，负责任务创建、审批、分发、聚合统计和中心侧目标清单。 |
| 边缘 API | 接收中心下发的任务包，向中心提供本 regionId 的统计和设备明细查询。 |
| 边缘执行进程 | 评估设备、下发 OTA 指令、更新边缘设备明细、消费 MQ 和 Redis 事件。 |
| 产品线网关 | 维护设备 Redis 状态，上报上线和 OTA 进度，接收 OTA 指令。 |
| 审批系统 | 审批指定版本号升级和文件导入升级任务。 |

核心规则：中心不保存设备执行明细。设备进度、成功、失败、步骤数据和失败原因保存在边缘。前端只访问中心，中心按用户选择的 regionId 转发到对应边缘查询明细。

## 大区与 Region 模型

一个中心 API 只管理一个大区。当前中心所属大区由顶层 `regionId` 推导，不属于当前大区的 `node_api` 节点会被过滤。

| 大区 | 管理的 regionId |
| --- | --- |
| 中国 `1` | 杭州 `1`、杭州低功耗 `150`、深圳 `100`、成都 `101`、上海 `160` |
| 亚洲 `2` | 亚洲 `2` |
| 北美 `3` | 北美 `3` |
| 欧洲 `4` | 欧洲 `4` |
| 开发模拟 `5` | 按中国大区处理 |

产品影响：

- 用户只能在当前大区中心内操作当前大区范围。
- 文件导入和手动导入不应让用户选择集群或节点。
- 指定版本号升级/活跃设备扫描可以让用户选择一个或多个 regionId，或选择整个大区。
- 已知属于其他大区的设备不应进入当前中心任务。

## 数据边界

| 存储 | 主要数据 |
| --- | --- |
| 中心 MongoDB | `UpgradeTask`、审批状态、`targetRegionIds`、`regionStatus`、`DeviceStatsSnapshot`、导入/手动目标清单、中心策略。 |
| 边缘 MongoDB | 边缘任务副本、边缘策略快照、本 regionId 的设备执行明细。 |
| Redis Stack DB 0 | `did:{did}`、`idx:device`、OTA 运行态、任务执行去重。 |
| RabbitMQ | 上线/状态队列和 OTA 进度队列。 |

中心侧统计是区域级聚合，例如 `total`、`pending`、`success`、`failed`、`completed`。中心不逐条写入设备进度。

## 任务创建与分发

不同 `deviceFrom` 的创建和分发方式不同：

| 任务类型 | 区域选择 | 目标来源 | 审批 | 分发时机 |
| --- | --- | --- | --- | --- |
| 活跃设备扫描/指定版本号升级 | 用户选择当前大区下的 regionId 范围。 | 边缘按机型和策略扫描活跃设备。 | 需要 | 审批通过后。 |
| 文件导入 | 用户不选择节点或 regionId。 | 中心通过日志平台补充机型/版本，通过设备归属接口补充 regionId。 | 需要 | 审批通过后。 |
| 手动导入 | 用户不选择节点或 regionId。 | 同文件导入。 | 不需要 | 发布后立即分发。 |

规则：

- 文件/手动导入的目标设备机型和旧版本来自创建阶段查询到的日志平台数据。
- 设备归属接口返回 `did/regionId`；OTA 不使用 `cluster`。
- 创建任务阶段不最终判断设备机型是否匹配，最终以边缘评估读取到的 `did:{did}.model/ver` 为准。
- 已知归属但属于其他大区的设备，不进入当前中心目标清单。
- 未知归属设备以空 `regionId` 保存，并在当前大区广播为 `status=-1` 的候选记录。
- 文件/手动任务如果没有任何当前中心可下发的目标设备，发布失败。
- 手动导入分发失败时，任务回滚到待发布。
- 审批任务分发失败时，任务置为失败。
- 审批回调必须幂等处理。只有审批中的任务能被审批通过、拒绝或置为过期。

## 候选认领

只有创建时未知归属的设备会进入候选认领流程：

1. 产品线网关把设备当前状态、版本、机型、类型写入 Redis 的 `did:{did}`。
2. 产品线网关发布设备上线事件。
3. 边缘消费上线事件，调用中心 `claimDevice(taskId, did, regionId, onlineTs)`。
4. 中心只在 `onlineTs` 更新且任务仍可认领时接受认领。
5. 认领成功后，候选记录从 `status=-1` 改为待升级 `status=0`。
6. 边缘把设备推入评估队列。

任务过期或已中止后，不应继续接受候选认领。

## 边缘执行规则

- 边缘收到分发后写入 `EdgeUpgradeTask` 和 `EdgeUpgradePolicy`。
- 活跃设备扫描任务按 `model` 入队；评估时读取 Redis Hash，只保留当前在线设备，即 `status=1`。
- 文件/手动导入的目标设备必须进入评估队列。
- 评估队列消息必须携带 `taskId`。
- 文件/手动设备优先使用目标清单里的 `versionOld`；没有时再读取 Redis 的 `did:{did}.ver`。
- 如果文件/手动设备由版本或 Redis 推导出的机型不属于当前任务策略，边缘直接写失败，原因是“设备机型不匹配”。
- 如果策略没有地区、国家、省市、经销商等条件过滤，评估时不查询日志平台，只依赖 Redis 版本、机型、在线态和策略版本规则。
- 如果策略有地区、国家、省市、经销商等条件过滤，评估时需要查询日志平台补充条件资料。
- 文件/手动导入设备当前离线时，应保持待升级运行态，等设备上线后第一时间下发。
- 同一机型多个任务可以同时存在，因为缓存、策略、数量和统计都按 `taskId` 隔离。
- 设备级互斥仍然成立：同一设备不能同时进入多个 OTA 下发链路。
- 同一任务内，设备只要已经进入过下发链路，失败后再次上线也不应重复创建新一轮下发。

## 任务隔离与数量上限

| 概念 | 规则 |
| --- | --- |
| 任务隔离 | 策略、缓存、评估消息、统计和数量限制都按 `taskId` 隔离。 |
| 设备互斥 | 同一设备不能同时进入多个 OTA 下发链路。 |
| 活跃扫描冲突 | 同一设备命中多个活跃扫描任务时，按当前流程描述执行最新匹配任务。 |
| 固定设备任务 | 文件/手动导入设备按各自明确的 `taskId` 评估。 |
| `max_upgrade_num` | 生效范围是 `taskId + model`。 |
| `single_max_upgrade_num` | 生效范围是 `taskId + model + versionOld`。 |

## 进度与统计

- 设备或网关把 OTA 进度、成功、失败发送到 `otaTaskProgressQueue`。
- 边缘更新 `EdgeUpgradeTaskDevice.stepData/status`。
- 中心只接收低频同步或生命周期同步，不接收每条设备进度。
- 详情页刷新时，前端请求中心，中心再向边缘查询当前区域聚合和设备明细。
- 如果实时边缘明细查询失败，应返回查询错误并保留上次统计，不应把查询失败当成设备升级失败。
- 文件/手动任务未执行前，列表总目标数可以来自中心目标清单；执行进度仍以边缘聚合为准。

## 中止语义

OTA2.0 技术方案中的中止流程如下：

1. 前端调用中心 `abortTask(taskId)`。
2. 中心将任务置为中止/失败状态，并把 `deadline` 设为当前时间。
3. 中心通知目标边缘 region。
4. 边缘把待升级和升级中设备置为失败。
5. 边缘清理候选、运行态和执行去重。
6. 边缘同步任务状态给中心。

重要产品提醒：早期操作指南说“结束任务后，已开始升级的设备继续完成”；OTA2.0 技术文档说“中止会把待升级/升级中设备置为失败”。写需求、手册、评审或测试时，必须确认当前采用哪一种最新产品规则，不能混用“结束任务”和“中止任务”语义。

## 设备规模快照

- 边缘通过 Redis Stack 的 `idx:device` 聚合生成低频设备规模快照。
- 不做 Node 侧全量 `SCAN did:*`。
- 快照按 `model/type` 聚合。
- `status` 变化频繁，不建索引。
- `ver` 不建索引，版本分布不进入每日快照。
- 中心只保存摘要，不保存完整 `model -> count` 大对象。摘要包括 total、type/model 数量、产品线汇总、Top 10 机型、其他机型汇总、扫描数量和是否完成。

## 产品与测试评审重点

OTA2.0 场景下必须检查：

- 需求是否遵守中心-边缘数据边界。
- 页面是否说明设备明细按 regionId 从边缘查询。
- 文件/手动导入是否由后台识别设备归属，而不是让用户选择节点。
- 未知归属设备是否通过候选认领处理。
- 是否同时覆盖 taskId 隔离和设备级互斥。
- `max_upgrade_num` 和 `single_max_upgrade_num` 的生效范围是否正确。
- 活跃扫描、文件导入、手动导入的边缘执行行为是否区分清楚。
- 中心聚合查询失败是否与设备升级失败区分清楚。
- “结束/中止任务”语义是否与最新确认的产品规则一致。
