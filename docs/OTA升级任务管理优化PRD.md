# OTA 升级任务管理优化 PRD

## 1. Executive Summary

### Problem Statement

当前 OTA 升级任务创建、任务列表和任务详情中，存在升级方式、任务状态、设备统计口径混用的问题。尤其在“指定版本号升级”场景下，设备是在任务执行过程中动态匹配的，创建任务时无法准确统计最终设备总数，若直接展示固定“升级设备总数”或固定百分比，会误导用户判断任务规模和执行进度。

### Proposed Solution

优化 OTA 升级任务创建流程、任务列表查询与字段、任务详情信息架构与统计口径展示。系统需明确区分“文件导入”“手动导入”“指定版本全量”“指定版本批量”四类统计口径，并在详情页支持状态模拟和统计口径模拟，方便产品、研发、测试验证不同业务场景下的展示逻辑。

### Success Criteria

- 指定版本升级场景下，不再将未知最终设备数展示为固定升级设备总数。
- 详情页中“任务流程进度”和“设备执行结果”口径清晰分离。
- 详情页不展示设备进度百分比仪表盘。
- 详情页可通过模拟按钮查看 7 种任务状态与 4 种统计口径组合。
- 任务列表字段、状态、操作按钮符合统一规则。
- 测试验收中，各状态和统计口径下无重复、缺失或误导性字段。

## 2. User Experience & Functionality

### User Personas

- **运营/实施用户**：创建 OTA 升级任务，查看任务状态与升级结果。
- **产线负责人**：审批 OTA 任务，判断升级范围、目标版本、风险和策略条件。
- **产品负责人**：确认任务创建、审批、执行、结束等流程是否闭环。
- **研发人员**：根据 PRD 实现页面字段、状态流转和统计口径逻辑。
- **测试人员**：验证不同任务状态、升级方式、统计口径下的页面展示是否正确。

### User Stories

- As a 运营用户, I want to step through task creation so that I can avoid missing required fields.
- As a 产线负责人, I want to see upgrade strategy and scope clearly so that I can approve tasks safely.
- As a 用户, I want to distinguish task flow progress from device execution result so that I do not misunderstand execution results.
- As a 研发人员, I want clear statistic rules so that I can implement dynamic matching scenarios correctly.
- As a 测试人员, I want status and statistic mode simulation buttons so that I can validate every display scenario efficiently.

### Acceptance Criteria

- 新增任务流程必须按步骤执行：基础信息 > 配置升级策略 > 预览发布 > 完成。
- 每一步进入下一步前必须校验必填项，并展示对应错误提示。
- 指定版本升级不再使用“全部版本升级 / 仅指定版本升级 / 排除指定版本不升级”切换，统一使用表格勾选源版本。
- 配置升级数量为“全量”时，设备数展示“全量”；配置为“批量”时，只支持统一输入计划成功下发数量。
- 任务列表查询区需支持任务名称、任务状态、升级方式、升级包、创建人、创建时间筛选。
- 任务列表需展示名称、升级方式、升级包、目标版本、升级设备数、任务时间、执行结果、任务所属大区、状态、创建人、创建时间、操作。
- 详情页主 Tab 只保留“概览”和“设备明细”。
- 详情页概览中保留“任务进度 / 流转明细”局部切换。
- 详情页提供状态模拟：待审批、已驳回、已失效、待执行、升级中、已完成、已结束。
- 详情页提供统计口径模拟：文件导入、手动导入、指定版本全量、指定版本批量。
- 指定版本全量执行中，执行概览展示“已匹配数 N 台”，不展示未知总数百分比。
- 指定版本全量完成后，提示为“升级成功 N 台，失败 N 台”。
- 指定版本批量执行中，可展示“已匹配数 / 计划成功下发数量”。

### Non-Goals

- 不实现真实后端设备检索、动态匹配和下发逻辑。
- 不实现真实审批流接口。
- 不实现真实文件上传解析和设备校验。
- 不实现跨大区查询，任务所属大区仍依赖顶部大区切换。
- 不新增 AI 能力。

## 3. AI System Requirements

不适用。本需求不涉及 AI 模型、AI 工具调用、生成式输出或 AI 评估策略。

## 4. Technical Specifications

### Architecture Overview

当前原型以静态 HTML/CSS/JavaScript 实现，核心模块包括：

- 新增任务向导
- 任务列表
- 任务详情
- 状态模拟
- 统计口径模拟
- 任务进度与流转明细
- 执行概览与异常分类

前端状态建议维护：

| 状态字段 | 说明 |
| --- | --- |
| `detailStatus` | 当前任务状态 |
| `detailMetricMode` | 当前统计口径 |
| `detailTab` | 当前详情主 Tab |
| `flowTab` | 当前任务进度局部 Tab |
| `createStep` | 当前新增任务步骤 |
| `quantityMode` | 全量 / 批量 |
| `taskFilters` | 任务列表查询条件 |
| `visibleTaskColumns` | 任务列表列设置 |
| `selectedVersions` | 选中的源版本 |
| `selectedRegions` | 选中的任务大区或策略地区 |

### Integration Points

后端接口待定，建议包含：

- 创建任务接口
- 保存草稿接口
- 提交审批接口
- 查询任务列表接口
- 查询任务详情接口
- 查询设备明细接口
- 查询流转明细接口
- 下载异常明细接口
- 结束任务接口
- 复制重建接口

建议数据字段：

| 字段 | 说明 |
| --- | --- |
| `taskName` | 任务名称 |
| `targetVersion` | 目标版本 |
| `upgradeMethod` | 指定版本 / 文件导入 / 手动导入 |
| `packageType` | 整包 / 差分包 |
| `region` | 任务所属大区 |
| `quantityMode` | 全量 / 批量 |
| `plannedSuccessCount` | 计划成功下发数量 |
| `matchedCount` | 已匹配数 |
| `upgradeSuccessCount` | 升级成功数 |
| `upgradeFailedCount` | 升级失败数 |
| `dispatchFailedCount` | 下发失败数 |
| `sourceVersions` | 指定版本升级的源版本列表 |
| `sourceScope` | 文件导入/手动导入的设备来源或指定版本源版本范围 |
| `condition` | 匹配条件 |
| `taskStatus` | 任务状态 |
| `createdBy` | 创建人 |
| `createdAt` | 创建时间 |
| `submittedAt` | 提交审批时间 |
| `approvedAt` | 审批通过 / 驳回时间 |
| `startAt` | 任务开始时间 |
| `endAt` | 任务结束时间 |

### Security & Privacy

- 设备 ID、区域信息、固件版本号属于业务敏感数据，前端不得在非授权页面泄露。
- 文件导入/手动导入设备清单需校验权限后展示。
- 审批、结束任务、删除草稿、复制重建等操作需后端权限校验。
- 操作审计记录需保留后端链路，但详情页主 Tab 不单独展示“操作记录”。

## 5. Risks & Roadmap

### Phased Rollout

#### MVP

- 完成新增任务流程。
- 完成任务列表字段和状态操作规则。
- 完成任务详情页概览、设备明细、任务进度、流转明细。
- 支持状态模拟和统计口径模拟。
- 明确文件导入、手动导入、指定版本全量、指定版本批量四类统计口径。

#### v1.1

- 接入真实任务列表与任务详情接口。
- 接入审批流状态。
- 接入设备明细与流转明细接口。
- 支持下载异常明细。

#### v2.0

- 支持真实动态匹配统计。
- 支持任务异常分析和失败原因聚合。
- 支持按角色控制详情字段和操作权限。
- 支持更完整的任务复盘与报表能力。

### Technical Risks

- 指定版本动态匹配最终设备数不可预估，若后端仍返回固定总数，前端展示会再次产生误导。
- “下发失败”和“升级失败”必须严格区分，否则批量模式下计划名额统计会错误。
- 状态流转与审批流若不同步，可能导致任务列表和详情状态不一致。
- 任务详情字段过多时，页面仍可能产生信息分散，需要保持“顶部任务信息、中部流程、下部结果”的分层原则。

## 6. Detailed Functional Requirements

### 6.1 新增任务流程

新增任务流程固定为：

```text
基础信息 → 配置升级策略 → 预览发布 → 完成
```

基础信息必填字段：

- 任务名称
- 任务执行大区
- 目标固件版本号
- 任务起止时间
- 任务升级说明

校验规则：

- 必填项未填写时，不允许进入下一步。
- 任务起止时间不允许选择过去时间。
- 默认开始时间为当前时间后 5 分钟。
- 快捷时间支持：未来 7 天、未来 30 天、未来 90 天。

### 6.2 配置升级策略

升级包类型：

- 整包
- 差分包

升级方式：

- 指定版本号升级
- 文件导入升级
- 手动导入升级

### 6.3 指定版本号升级

交互要求：

- 不再使用“全部版本升级 / 仅指定版本升级 / 排除指定版本不升级”切换。
- 使用源版本表格勾选。
- 表格字段包括：
  - 源版本号
  - 升级设备数

配置升级数量：

- 全量：表格展示“全量”。
- 批量：用户只输入统一计划成功下发数量。

说明文案：

- 全量：`按规则持续匹配符合条件设备，实际设备数以执行结果为准`
- 批量：`系统按计划成功下发数量控制名额，并持续匹配符合条件设备`

### 6.4 预览发布

预览发布页不再单独展示重复的“任务预览”信息块，避免用户将配置摘要误认为最终设备统计。

预览发布页保留：

- 升级策略
- 预检结果
- 异常明细下载
- 发布/提交审批操作

升级策略标题下方需展示策略说明：

- 指定版本全量：`按规则持续匹配符合条件设备，实际设备数以执行结果为准。`
- 指定版本批量：`系统按计划成功下发数量控制名额，并持续匹配符合条件设备。`
- 文件导入/手动导入：`设备清单固定，实际执行结果以设备回传为准。`

指定版本源版本表格字段：

- 源版本号
- 升级设备数

展示规则：

- 全量：升级设备数展示 `全量`。
- 批量：升级设备数展示用户设置的计划数量。
- 表头不得使用“口径”“计划”等额外后缀，统一保留“升级设备数”。

### 6.5 文件导入升级

功能要求：

- 支持 CSV / Excel。
- 上传前不展示预检结果。
- 上传后展示文件名、识别设备数量、上传完成状态。
- 不向用户展示研发校验规则。
- 支持重新上传。

### 6.6 手动导入升级

功能要求：

- 最多 10 台。
- 字段包括：
  - 设备 ID
  - 源固件版本号
  - 所属大区
  - 校验状态
  - 异常说明
  - 操作

## 7. Task List Requirements

### 7.1 查询区

查询区需采用单行优先布局，空间不足时允许自然换行，但控件顺序保持稳定。

查询条件顺序：

1. 任务名称
2. 任务状态
3. 升级方式
4. 升级包
5. 创建人
6. 创建时间

字段规则：

| 条件 | 默认值 | 交互方式 | 查询规则 |
| --- | --- | --- | --- |
| 任务名称 | 空 | 输入框 | 支持任务名称模糊搜索 |
| 任务状态 | 全部 | 下拉单选 | 支持全部、待发布、待审批、待执行、升级中、已完成、已结束、已驳回、已失效 |
| 升级方式 | 全部 | 下拉单选 | 支持全部、指定版本、文件导入、手动导入 |
| 升级包 | 全部 | 下拉单选 | 支持全部、整包、差分包 |
| 创建人 | 空 | 输入 + 下拉选择 | 支持模糊搜索创建人并选择 |
| 创建时间 | 空 | 日期选择器 | 选择某日创建的任务，不使用时间区间 |

交互要求：

- 查询条件变更后，列表需真实过滤并重置到第一页。
- 创建人输入时展示匹配候选项，点击候选项后填入输入框并触发筛选。
- 任务名称仅查询任务名称，不再聚合查询任务 ID 和目标版本。
- 控件需左对齐，字段之间保持一致间距。

### 7.2 列表字段与布局

任务列表字段：

- 名称
- 升级方式
- 升级包
- 目标版本
- 升级设备数
- 任务时间
- 执行结果
- 任务所属大区
- 状态
- 创建人
- 创建时间
- 操作

排序规则：

- 默认按创建时间倒序。

列设置：

- 支持用户勾选展示字段。
- 操作列固定。
- 文案过长时支持横向滚动。

工具栏：

- 新增任务按钮作为主按钮放置在列表右上。
- 列设置、刷新、紧凑按钮作为次级工具按钮分组展示。
- 工具按钮使用图标优先表达，文本只保留必要标签。

字段对齐：

- 表头和单元格统一左对齐。
- “升级设备数”列不展示“计划”“台”等冗余后缀。
- 指定版本全量展示 `全量`。
- 指定版本批量展示对应设置数量。
- 文件导入/手动导入展示固定设备数量。

### 7.3 状态操作

状态操作规则：

| 状态 | 操作 |
| --- | --- |
| 待发布 | 编辑、删除 |
| 待审批 | 详情 |
| 待执行 | 详情 |
| 升级中 | 详情、结束任务 |
| 已完成 | 详情 |
| 已结束 | 详情 |
| 已驳回 | 详情、复制重建 |
| 已失效 | 详情、复制重建 |

## 8. Task Detail Requirements

### 8.1 页面结构

详情页主 Tab：

```text
概览 / 设备明细
```

概览页结构：

```text
任务信息
任务进度
执行概览
异常分类
```

### 8.2 任务信息卡

顶部任务信息卡关注升级任务本身，而不是任务流程进度或设备执行百分比。

展示字段：

- 任务名称
- 目标版本
- 任务时间
- 任务大区
- 创建人
- 升级方式
- 升级包
- 升级设备数
- 下发方式
- 指定源版本
- 匹配条件
- 设备来源
- 任务说明
- 当前状态

字段展示规则：

- 不展示“升级配置摘要”这类汇总字段，需拆分为明确字段，便于用户逐项查阅配置。
- 指定版本全量：升级设备数展示 `全量`。
- 指定版本批量：升级设备数展示计划数量。
- 文件导入/手动导入：升级设备数展示固定设备数量。
- 指定版本场景必须展示“指定源版本”，以标签形式列出创建任务时选择的源版本。
- 指定版本场景必须展示“下发方式”：
  - 全量：`动态匹配，执行期间持续匹配符合条件设备`
  - 批量：`批量下发，按计划成功下发数量控制`
- 文件导入/手动导入场景展示“设备来源”，说明设备来自文件清单或手动录入。
- 匹配条件需展示地区等限制条件，并说明设备当前版本命中任一指定源版本后纳入匹配。

### 8.3 设备执行结果口径

详情页不展示升级进度百分比仪表盘，不展示会让用户误以为存在固定分母的百分比。

执行结果表达的是设备执行口径，不表达任务流程阶段。

执行类状态包括：

- 升级中
- 已完成
- 已结束

| 统计口径 | 展示规则 |
| --- | --- |
| 文件导入 | 展示固定设备总数、升级成功、升级失败、进行中/未处理 |
| 手动导入 | 展示固定设备总数、升级成功、升级失败、进行中/未处理 |
| 指定版本全量 | 展示已匹配数、升级成功、升级失败、匹配状态，不展示未知总数百分比 |
| 指定版本批量 | 展示已匹配数 / 计划成功下发数量 |

指定版本全量场景：

- 升级中：展示 `已匹配 N 台，持续动态匹配符合条件设备`
- 已完成：展示 `已匹配 N 台，匹配已结束`
- 已结束：展示 `已匹配 N 台，已停止继续匹配`

### 8.4 任务进度模块

局部 Tab：

- 任务进度
- 流转明细

任务进度只表达流程阶段，不展示设备统计百分比、仪表盘或执行结果占比。

阶段包括：

- 创建任务
- 提交审批
- 审批结果
- 等待执行
- OTA 下发
- 任务结束

阶段节点图标：

| 阶段 | 图标语义 |
| --- | --- |
| 创建任务 | 文档 |
| 提交审批 | 用户 |
| 审批结果 | 盾牌 |
| 等待执行 | 时钟 |
| OTA 下发 | 刷新 |
| 任务结束 | 完成 |

状态文案规则：

- 待审批：当前节点停留在“审批结果”，提示审批处理中，任务尚未进入执行队列。
- 已驳回：当前节点停留在“审批结果”，展示驳回原因；后续执行节点置灰。
- 已失效：当前节点停留在“审批结果”，展示失效原因；后续执行节点置灰。
- 待执行：当前节点停留在“等待执行”，文案为等待到达计划开始时间，不得描述为“已到达开始时间”。
- 升级中：当前节点停留在“OTA 下发”，提示执行结果持续回传中。
- 已完成：所有流程节点完成。
- 已结束：展示为中性终止态，说明任务提前结束和结束原因；不得使用与异常失败完全相同的错误态表达。

流转明细规则：

- 流转明细按时间顺序展示创建、提交审批/发布、审批结果、开始下发、执行中/完成/结束。
- 待执行状态不得记录为“已开始下发”，应展示为“等待计划开始时间”。
- 已驳回、已失效状态不展示 OTA 下发和任务完成记录。

### 8.5 执行概览

非执行类状态展示“任务配置摘要”，不展示执行结果：

- 待审批：展示升级设备数和当前节点。
- 已驳回：展示升级设备数和驳回原因。
- 已失效：展示升级设备数和失效原因。
- 待执行：展示升级设备数和计划开始时间。

执行类状态展示“执行概览”。

文件导入/手动导入：

- 升级设备总数
- 升级成功
- 升级失败
- 进行中 / 未处理

指定版本全量：

- 已匹配数
- 升级成功
- 升级失败
- 匹配状态

指定版本批量：

- 计划成功下发数量
- 已匹配数
- 升级失败
- 待匹配名额

### 8.6 异常分类

仅当存在异常设备时展示异常分类；若异常设备数为 0，则不展示该模块。

执行类状态且存在异常设备时展示：

- 设备离线或长时间未上报
- 升级超时或设备主动失败

支持下载异常明细。

### 8.7 设备明细

设备明细主 Tab 展示进入执行链路后的设备明细。

展示规则：

- 待审批、已驳回、已失效、待执行：任务尚未进入执行阶段，展示空状态，不展示设备表格。
- 升级中、已完成、已结束：展示设备明细表格。
- 指定版本全量：设备明细只展示已动态匹配并进入下发链路的设备；已结束时可展示未继续下发设备的终止状态。
- 指定版本批量：设备明细展示已匹配、已下发、升级结果和剩余名额相关状态。
- 文件导入/手动导入：设备明细展示固定设备清单。

表格字段：

- 设备 ID
- 源版本
- 目标版本
- 所属大区
- 下发状态
- 升级状态
- 完成时间
- 失败原因
- 最近上报时间

交互要求：

- 设备 ID 搜索框需要真实过滤设备明细；若原型阶段不实现过滤，则不得使用“按 Enter 搜索”等强交互文案。

## 9. Statistic Mode Rules

统计口径模拟按钮：

```text
文件导入 / 手动导入 / 指定版本全量 / 指定版本批量
```

### 文件导入

- 设备总数已知。
- 分母为设备清单总数。
- 展示固定设备总数、升级成功、升级失败、进行中/未处理。

### 手动导入

- 设备总数已知。
- 分母为手动录入设备总数。
- 展示固定设备总数、升级成功、升级失败、进行中/未处理。

### 指定版本全量

- 设备执行中动态匹配。
- 不展示最终总数百分比。
- 展示已匹配数。
- 可展示已匹配设备中的成功/失败数量和占比。

示例：

```text
已匹配数：358 台
升级成功：346 台，占已匹配 96.6%
升级失败：12 台，占已匹配 3.4%
```

### 指定版本批量

- 有计划成功下发数量。
- 展示已匹配数 / 计划成功下发数量。
- 展示待匹配名额。
- 系统按计划成功下发数量控制名额，并持续匹配符合条件设备。

示例：

```text
计划成功下发数量：5,000 台
已匹配数：358 / 计划 5,000 台
升级成功：346 台，占已匹配 96.6%
升级失败：12 台，占已匹配 3.4%
```

## 10. Task Status Rules

| 状态 | 展示重点 |
| --- | --- |
| 待审批 | 展示配置摘要、审批等待文案，不展示升级执行结果 |
| 已驳回 | 展示驳回原因，不展示升级执行结果 |
| 已失效 | 展示失效原因，不展示升级执行结果 |
| 待执行 | 展示计划开始时间，不展示升级执行结果 |
| 升级中 | 展示执行概览、已匹配数、成功/失败数量 |
| 已完成 | 展示最终成功/失败结果 |
| 已结束 | 展示已匹配数、成功/失败数量、结束原因 |

## 11. Testing & Acceptance

核心验收用例：

- 创建任务每一步必填校验生效。
- 指定版本全量下，详情页不展示未知总数百分比。
- 指定版本全量升级中，执行概览显示“已匹配数 N 台”。
- 指定版本全量完成后，提示为“升级成功 N 台，失败 N 台”。
- 指定版本批量下，展示“计划成功下发数量”和“待匹配名额”。
- 文件导入/手动导入下，展示明确设备总数和执行结果，不展示百分比仪表盘。
- 主详情 Tab 不展示“操作记录”。
- 任务进度内保留“流转明细”。
- 任务列表不同状态下操作按钮符合规则。
- “刷新 | 进度报告”不在详情页任务进度模块展示。
- “任务大区”字段展示区域图标。
- 阶段节点使用语义化图标，不使用纯数字作为默认节点表达。

## 12. Open Questions

- 后端是否能区分“下发失败”和“升级失败”两个字段。
- 指定版本批量中的计划成功下发数量是否以“成功匹配”还是“成功下发”作为最终占用名额。
- 指定版本全量完成后的实际总数是否等于最终已匹配数。
- 文件导入和手动导入是否统一使用同一统计口径字段。
- 审批超时失效的具体时间规则是否由审批系统或 OTA 系统计算。
